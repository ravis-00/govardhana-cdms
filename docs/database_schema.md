# Govardhana CDMS – Database Schema

## Current Version

**v2.3-preventive-care**

## Last Updated

23-Jul-2026

---

## 1. General Rules

1. Row 1 of every operational Google Sheet contains column headers.
2. Header names should use lowercase `snake_case`.
3. Apps Script should map values using header names wherever possible.
4. IDs must be generated by Apps Script unless explicitly documented otherwise.
5. IDs should not depend on spreadsheet formulas.
6. Cattle identifiers and tag numbers must be written as plain text.
7. Display dates use `DD-MM-YYYY`.
8. API date values use `YYYY-MM-DD`.
9. Timestamps should include both date and time.
10. Historical records should not be overwritten when an event history is required.

# 2. `cattle_master`

## Purpose

Permanent master record for every cattle.

## Primary Key

`internal_id`

## Important Rule

`tag_number` stores only the current active tag.

Historical tags are stored in `tag_history`.

## Columns

| Column | Purpose |
|---|---|
| internal_id | Permanent cattle ID generated by the backend |
| tag_number | Current active ear-tag number |
| govt_uid | Government-issued cattle UID |
| prev_tag_numbers | Previous or imported tag references where retained |
| cattle_name | Name of cattle |
| gender | Female or Male |
| category | Cow, Heifer, Bull, Ox, Female Calf, Male Calf, etc. |
| breed | Breed of cattle |
| color | Base colour |
| dob | Date of birth |
| shed_id | Assigned shed |
| status | Active, exited, dead or other lifecycle status |
| adoption_status | Sponsorship/adoption status |
| photo_url | Cattle photo URL |
| is_disabled | Disability flag |
| disability_details | Disability description |
| remarks | General remarks |

## Rules

- `internal_id` must be unique.
- `tag_number` must be unique among current cattle records.
- `internal_id` must never change.
- Historical tag numbers must not replace the permanent identity.
- Deactivated or exited cattle remain in the master for traceability.

# 3. `cattle_origins`

## Purpose

Store admission and origin details for every registered cattle.

## Key

`internal_id`

## Columns

| Column | Purpose |
|---|---|
| internal_id | Links to `cattle_master.internal_id` |
| admission_date | Date cattle entered the system |
| admission_type | Born at Goshala, Purchase, Donation, From Farmer, Rescue / Slaughter House |
| admission_age_months | Age at admission in total months |
| source_party_name | Source person, farmer, donor, seller or organisation |
| source_party_address | Source address |
| source_party_mobile | Source phone number |
| admission_weight | Weight at admission |
| birth_weight | Weight at birth where applicable |
| purchase_price | Purchase price for Purchase admission |
| dam_id | Mother internal ID |
| dam_breed | Mother breed |
| sire_id | Father internal ID |
| sire_breed | Father breed |
| linked_birth_id | Linked `birth_log.transaction_id` for Born-at-Goshala registrations |

## Rules

- One active origin record should exist for each registered cattle.
- `linked_birth_id` is required for Born-at-Goshala registrations.
- Source-party fields depend on admission type.
- Purchase price applies only to Purchase admission.
- Dam and sire information should be preserved for born-at-Goshala cattle.

# 4. `birth_log`

## Purpose

Maintain birth and calving records before calf registration.

## Primary Key

`transaction_id`

## Columns

| Column | Purpose |
|---|---|
| transaction_id | Unique birth transaction ID |
| date_of_birth | Date of birth |
| time_of_birth | Time of birth |
| mother_tag_input | Mother tag entered by user |
| mother_id | Mother internal ID |
| mother_breed | Mother breed |
| father_tag_input | Father tag entered by user |
| father_id | Father internal ID where available |
| father_source | Own, Borrowed or Unknown |
| father_owner | Owner/source of borrowed father |
| father_breed_manual | Manually entered father breed |
| father_breed_final | Final resolved father breed |
| calf_breed | Calculated calf breed |
| calf_sex | Female or Male |
| calf_weight | Birth weight |
| calf_colour | Calf colour |
| delivery_type | Normal or other delivery classification |
| disability_details | Disability details |
| birth_status | Healthy, Weak, Abortion or Stillborn |
| remarks | Birth remarks |
| photo_url | Newborn photo URL |
| status | Pending, Registered or other workflow status |

## Rules

- `transaction_id` must be unique.
- Mother must be a valid eligible female cattle.
- Mother must be Active.
- Mother-age and calving-gap rules must be validated.
- Calf breed is derived from parent breeds.
- Abortion and Stillborn records are excluded from registration.
- Eligible live calves may be registered after 21 days.
- Registration updates `status` to Registered.
- Registered birth ID is stored in `cattle_origins.linked_birth_id`.

# 5. `tag_history`

## Purpose

Maintain complete tag-change traceability.

## Primary Key

`history_id`

## Columns

| Column | Purpose |
|---|---|
| history_id | Unique tag-history transaction ID |
| internal_id | Permanent cattle ID |
| old_tag_number | Previous tag number |
| new_tag_number | New active tag number |
| change_date | Date of tag change |
| reason | Reason for changing tag |
| changed_by | Person who changed the tag |
| remarks | Additional remarks |

## Rules

- `history_id` must be unique.
- `internal_id` must refer to a cattle record.
- `new_tag_number` becomes the current tag in `cattle_master`.
- Existing tag-history rows must not be overwritten.
- Tag-history records should be sorted latest first in the UI.

|---|---|
| exit_id | Unique lifecycle-event ID |
| internal_id | Permanent cattle ID |
| tag_number | Tag number at the time of event |
| exit_date | Date of exit or reactivation |
| exit_time | Time of event |
| exit_type | Sale, Transfer, Farmer Handover, Death, Reactivation or other approved value |

| remarks | Additional remarks |

| teeth_age | Estimated age from teeth |

# 7. `health_log`

## Purpose

Store Clinical Records and treatment history.

## Primary Key

`transaction_id`

## ID Format

`MED-#####`

Examples:

- `MED-00001`
- `MED-00002`

## Columns

| Column | Purpose |
|---|---|
| transaction_id | Permanent clinical-record ID |
| treatment_date | Date of clinical visit |
| internal_id | Cattle identifier used for the treatment record |
| weight_at_treatment | Weight at visit; optional and not yet fully implemented |
| symptom | Selected symptoms and diagnoses stored as comma-separated text |
| diagnosis_detail | Additional diagnosis information |
| medicine_given | Selected medicines stored as comma-separated text |
| dosage_details | Existing dosage text field |
| doctor_name | Treating doctor or veterinarian |
| treatment_cost | Optional treatment cost |
| photo_url | Treatment photo or attachment URL |
| remarks | Clinical remarks and treatment instructions |

## Rules

- `transaction_id` must be unique.
- IDs are generated by Apps Script.
- Spreadsheet formulas must not generate IDs.
- New Clinical Records require a valid Active cattle record.
- Historical records may be edited without changing their original cattle identity.
- Cattle IDs must be stored as plain text.
- Valid IDs follow `MED-#####`.
- Blank, malformed and timestamp IDs are ignored when calculating the next ID.

# 8. `preventive_care_log`

## Purpose

Store preventive-care campaigns and administration events.

## Primary Key

`event_id`

## ID Format

`PC-######`

Examples:

- `PC-000001`
- `PC-000002`

## Columns

| Column | Purpose |
|---|---|
| event_id | Unique preventive-care event ID |
| event_date | Administration date |
| care_type_id | ID from `preventive_care_types` |
| care_type_name | Care-type name stored for reporting stability |
| medicine_id | Medicine master ID where available |
| medicine_name | Medicine or product name |
| medicine_batch_no | Medicine batch number |
| medicine_expiry_date | Expiry date of the medicine batch |
| dosage | Numeric dosage |
| dosage_unit | ml, mg, gm, tablet, dose or other unit |
| administration_route | Oral, Intramuscular, Subcutaneous, Topical, etc. |
| target_group | Description of cattle group |
| eligible_count | Number eligible for the event |
| administered_count | Number actually administered |
| excluded_count | Eligible minus administered |
| next_due_date | Next preventive-care schedule |
| doctor_name | Doctor or administering person |
| status | Draft, Completed or Cancelled |
| remarks | Event remarks |
| created_by | User who created the record |
| created_at | Record creation timestamp |
| updated_by | User who last updated the record |
| updated_at | Last update timestamp |

## Rules

- `event_id` must be unique.
- IDs are generated by Apps Script.
- `care_type_id` and `care_type_name` are both stored.
- `medicine_id` and `medicine_name` are both stored where available.
- `eligible_count` must be greater than zero.
- `administered_count` cannot be negative.
- `administered_count` cannot exceed `eligible_count`.
- `excluded_count` is calculated by the backend.
- `next_due_date` cannot be earlier than `event_date`.
- Expired medicine cannot be recorded for an administration date after expiry.
- `created_by` and `created_at` are preserved during update.
- `updated_by` and `updated_at` are refreshed during update.

## Event Status Values

- Draft
- Completed
- Cancelled

## Purpose

Master list of preventive-care types.

## Primary Key

`care_type_id`

## ID Format

`PCT-###`

Examples:

- `PCT-001`
- `PCT-002`

## Columns

| Column | Purpose |
|---|---|
| care_type_id | Unique care-type ID |
| care_type_name | Display name |
| description | Description of care type |
| display_order | UI display order |
| is_active | Yes or No |
| remarks | Additional remarks |

## Standard Care Types

- Vaccination
- Deworming
- Vitamin Supplementation
- Mineral Supplementation
- Other Preventive Treatment

## Rules

- Care Type Name must be unique.
- Display Order must be a whole number between 1 and 99.
- Active values are stored as Yes or No.
- Inactive types should not appear in new-entry dropdowns.
- Historical records retain stored care-type names.
